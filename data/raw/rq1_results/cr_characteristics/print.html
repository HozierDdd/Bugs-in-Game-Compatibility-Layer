Analyze 2：Submission Prevalence 实验报告

R 语言可复现实验模板 · ProtonDB CR、Issue Tracker CR 与 comments 的月度 prevalence 分析
研究目标。 本报告以“游戏-月份”为面板单位，分析哪些游戏特征、更新、折扣、Proton 版本发布、Steam Deck 相关因素，与每月 CRs / comments 数量相关；进一步比较 ProtonDB 与 issue tracker 两个平台上的影响是否相同，并与 Lin et al. 的 Steam review prevalence 研究第 4.2 节的建模思路进行对照。

1. 研究背景与参考方法

Lin et al. 在 Steam 游戏评论研究第 4.2 节中，将“某游戏某一天收到的评论数”作为因变量，使用 mixed-effect model 分析游戏特征与每日评论数的关系。该节的核心流程包括：数据标准化、Pearson 相关性分析、冗余分析、混合效应模型构建，并将 game id、developer、studio size 作为随机效应，将 owners、players、early access、age、last update、last discount、last discount life 等作为固定效应。

本实验将该思路从 daily game reviews 扩展到 monthly CR / comment submissions。由于 CR 和 comments 都是计数型因变量，本报告建议以 negative binomial mixed-effects model 为主模型；若零值过多，则使用 zero-inflated negative binomial model；线性混合模型仅作为稳健性分析。

2. 研究问题与假设

RQ 问题 可检验假设 主要模型
RQ1 哪些因素与 ProtonDB 和 issue tracker 的月度 CR / comment prevalence 相关？ players、owners、更新、折扣、Proton release、Proton release mention、Steam Deck 时间变量与提交数量显著相关。 平台内负二项混合模型
RQ2 影响 CR prevalence 的因素是否与 Lin et al. 中影响 game review prevalence 的因素相似？ players / owners / discount / update 的方向可与 Lin et al. 第 4.2 节对照；CR 可能更受 Proton release 与 Steam Deck 相关变量影响。 标准化系数方向和 IRR 比较
RQ3 两个平台的影响因素是否相同？ ProtonDB 更受 Proton release 与 release note mention 影响；issue tracker 可能更受游戏更新和讨论活跃度影响。 platform × factors 交互模型

panel <- panel_raw %>%
  mutate(

panel <- panel %>%

5. 诊断与模型选择

5.1 相关性分析

cor_vars <- panel %>%
  select(starts_with("z_")) %>%
  select(where(~ sd(.x, na.rm = TRUE) > 0))

cor_mat <- cor(cor_vars, use = "pairwise.complete.obs", method = "pearson")

high_cor <- as.data.frame(as.table(cor_mat)) %>%
  filter(Var1 != Var2, abs(Freq) > 0.7) %>%
  arrange(desc(abs(Freq)))

print(high_cor)
变量保留规则。 如果两个变量 |r| > 0.7，只保留理论上更接近研究问题、缺失更少、解释更直接的变量。例如 discount_life 与 last_discount_life 高相关时，不建议无条件同时进入主模型。

5.2 冗余分析

fixed_formula <- as.formula(paste("~", paste(names(cor_vars), collapse = " + ")))
redun_result <- redun(fixed_formula, data = panel, nk = 0, r2 = 0.90)
print(redun_result)

5.3 过度离散与零膨胀检查

m_pois_cr <- glmmTMB(
  cr_count ~ z_log_players + z_log_owners + z_age + z_update_number +
    z_avg_discount + z_discount_life + z_last_update + z_last_discount +
    z_last_discount_life + z_log_game_size + z_proton_release +
    z_proton_release_number + proton_release_mention_game + z_last_mention +
    z_last_deck + (1 | game_id) + (1 | developer),
  family = poisson(),
  data = filter(panel, platform == "protondb")
)

check_overdispersion(m_pois_cr)
check_zeroinflation(m_pois_cr)

sim_pois <- simulateResiduals(m_pois_cr)
plot(sim_pois)
testDispersion(sim_pois)
testZeroInflation(sim_pois)
模型选择建议。 如果 Poisson 过度离散，主模型使用 family = nbinom2()；如果零值明显多于模型预期，使用 ziformula = ~ 1 或 ziformula = ~ platform 的 zero-inflated negative binomial model。

6. 模型构建

6.1 平台内模型：ProtonDB CR prevalence

m_nb_protondb_cr <- glmmTMB(
  cr_count ~ z_log_players + z_log_owners + z_age + z_update_number +
    z_avg_discount + z_discount_life + z_last_update + z_last_discount +
    z_last_discount_life + z_log_game_size + z_proton_release +
    z_proton_release_number + proton_release_mention_game + z_last_mention +
    z_last_deck + (1 | game_id) + (1 | developer),
  family = nbinom2(),
  data = filter(panel, platform == "protondb")
)

summary(m_nb_protondb_cr)
modelsummary(m_nb_protondb_cr, exponentiate = TRUE, stars = TRUE)

负二项模型中，exponentiate = TRUE 后的系数为 incidence rate ratio，即 IRR。例如 IRR = 1.20 表示自变量增加 1 个标准差时，月度 CR 期望数量增加约 20%。

6.2 平台内模型：Issue tracker CR 与 comments prevalence

m_nb_issue_cr <- update(
  m_nb_protondb_cr,
  data = filter(panel, platform == "issue_tracker")
)

m_nb_issue_comment <- glmmTMB(
  comment_count ~ z_log_players + z_log_owners + z_age + z_update_number +
    z_avg_discount + z_discount_life + z_last_update + z_last_discount +
    z_last_discount_life + z_log_game_size + z_proton_release +
    z_proton_release_number + proton_release_mention_game + z_last_mention +
    z_last_deck + (1 | game_id) + (1 | developer),
  family = nbinom2(),
  data = filter(panel, platform == "issue_tracker")
)

modelsummary(

7.3 与 Lin et al. 第 4.2 节的对照

因素 Lin et al. 第 4.2 节中对 daily review prevalence 的结论 本实验中的比较方式
players active players 是最强相关因素，强于 owners。 检查 z_log_players 是否仍是 CR / comments 的最大 IRR。
owners 显著，但效应弱于 players。 判断其是否反映潜在用户基数，而非即时提交行为。
last discount / discount life 折扣事件比更新更能带来 review surge。 判断折扣是否也刺激 CR，还是只刺激 review。
last update / update_number 在 review prevalence 中影响较弱。 CR / issue comments 可能更受更新影响，因为更新会改变兼容性状态。
Proton release / mention Lin et al. 未研究。 本实验核心新增变量，预期 ProtonDB 效应更强。
Steam Deck timing Lin et al. 未研究。 检查 Deck 发布后 CR 与 comments 是否出现结构性变化。

7.4 推荐结果表述

  • 正向效应：控制 game id 与 developer 的随机差异后，players 与月度 CR 数量呈正相关。IRR = X，95% CI [L, U]，表示月平均玩家数增加 1 个标准差时，期望 CR 数量增加约 (X - 1) × 100%。
  • 负向效应：last_update 的 IRR 小于 1，表示距离上次游戏更新越久，月度提交越少；换言之，更新后的近期更可能触发提交。
  • 跨平台差异：若 platformissue_tracker:z_proton_release_number 显著，说明 Proton release 次数对 issue tracker 的影响不同于 ProtonDB。

8. 效度威胁

类别 威胁 缓解方式
内部效度 CR 数量增长可能由未观测事件驱动，例如大型补丁、社区讨论、外部新闻。 加入月份固定效应或 year-month random intercept；进行敏感性分析。
构念效度 CR / comment prevalence 不等于真实兼容性问题 prevalence，只表示用户提交行为。 明确称为 submission prevalence，不解释为真实 bug 发生率。
统计结论效度 计数数据可能过度离散、零膨胀、同一游戏内自相关。 使用 negative binomial / zero-inflated model；做 DHARMa 残差诊断。
外部效度 结果可能只适用于 Steam / ProtonDB 生态。 讨论中限定研究范围；与 Lin et al. 研究只做平台内类比。
